# SQL学习笔记：窗口函数 vs GROUP BY 性能对比案例分析

## 一、测试背景

**测试目的**：对比窗口函数 + DISTINCT 与 GROUP BY 在相同聚合查询下的性能差异。

**测试场景**：查询每个大学的平均答题数。

**测试数据**：`user_profile` 表，包含 university、question_cnt 等字段。


## 二、测试结果

| 写法 | 运行时间 | 占用内存 | 结果 |
|------|---------|---------|------|
| 窗口函数 + DISTINCT | **34ms** | 6672KB | ✅ 更快 |
| GROUP BY | 47ms | 6552KB | 稍慢 |

**结论**：窗口函数版本比 GROUP BY 版本快 **28%**（47ms → 34ms）


## 三、两种写法的代码

### 写法1：窗口函数 + DISTINCT
```sql
SELECT
    DISTINCT university,
    AVG(question_cnt) OVER (PARTITION BY university) AS avg_question_cnt
FROM user_profile
ORDER BY avg_question_cnt;
```

### 写法2：GROUP BY（传统聚合）
```sql
SELECT
    university,
    AVG(question_cnt) AS avg_question_cnt
FROM user_profile
GROUP BY university
ORDER BY avg_question_cnt;
```


## 四、为什么窗口函数反而更快？

| 因素 | 数据特征 | 对窗口函数有利 | 对 GROUP BY 影响 |
|------|---------|---------------|-----------------|
| **分组数量** | 少（几个大学） | ✅ 去重开销小 | 哈希表小但仍有开销 |
| **每组数据量** | 大（每校多学生） | ✅ 流式计算高效 | 哈希聚合需维护 |
| **计算类型** | AVG（可维护状态） | ✅ 累加 + 计数即可 | 需要存储中间状态 |

**核心原因**：
- **窗口函数**：一次扫描，流式计算，最后对少量结果去重
- **GROUP BY**：需要建立哈希表，存储每个分组的聚合状态，有额外内存和计算开销


## 五、执行计划差异（理论分析）

```
GROUP BY 执行路径：
Table Scan → Hash Aggregate → Sort → Output
            ↑
         建哈希表（内存/磁盘）
         哈希冲突处理
         内存管理开销

窗口函数 + DISTINCT 执行路径：
Table Scan → Window Aggregate → Sort → Unique → Output
            ↑                ↑
         流式计算          小结果集去重
         （无哈希表）       （分组数少）
```

**关键差异**：GROUP BY 需要维护哈希表，窗口函数只需维护当前分组的累加状态。


## 六、这个测试的局限性

| 条件 | 说明 |
|------|------|
| **数据量** | 测试数据量较小（毫秒级） |
| **分组数** | 大学数量少（窗口函数的优势区） |
| **缓存影响** | 窗口函数先执行可能已缓存数据 |
| **MySQL 版本** | 8.0+ 对窗口函数有优化 |

**结论**：这个测试结果**不是普适结论**，只代表当前数据环境。


## 七、性能对比总结表

| 场景 | 窗口函数 | GROUP BY | 说明 |
|------|---------|----------|------|
| 分组少（<100），每组数据大 | ✅ 更快 | 较慢 | 窗口函数流式计算优势 |
| 分组多（>10000），每组数据小 | 较慢 | ✅ 更快 | DISTINCT 去重开销大 |
| 中等数据量 | 相近 | 相近 | 差异不明显 |
| 需要多个聚合值 | ✅ 一次扫描 | 可能需多次 | 窗口函数优势 |
| 只需简单聚合 | 可用 | ✅ 更清晰 | 语义优先 |


## 八、最终结论

| 问题 | 答案 |
|------|------|
| 窗口函数一定比 GROUP BY 慢吗？ | **不一定**，本测试窗口函数快 28% |
| 为什么窗口函数能更快？ | 分组少时，流式计算 + 小结果集去重，避免哈希表开销 |
| 窗口函数能替代 GROUP BY 吗？ | 能，但**不是主要用途** |
| 窗口函数的真正价值是什么？ | 排名、累计、环比、保留明细等 **GROUP BY 做不到的事** |
| 应该怎么选？ | **以实际数据测试为准**，不要凭感觉 |


## 九、一句话总结

> **窗口函数在"分组少、数据量大"的场景下性能可能优于 GROUP BY，但它的真正价值是做 GROUP BY 做不到的事（排名、累计、保留明细）。本测试仅证明性能差异存在，具体选哪个应以实测为准。**
